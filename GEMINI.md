# 🤖 마중(Majung) 프로젝트 AI 가이드라인 및 피그마 이식 지침서 (GEMINI.md)

이 문서는 **마중(Majung)** 플러터 프로젝트를 개발하는 모든 AI 코딩 어시스턴트가 반드시 숙지하고 철저히 준수해야 하는 행동 지침서이자 피드백 기록부입니다. 이전 개발 단계에서 발생한 오류 및 피그마 디자인 불일치 사례를 바탕으로 작성되었으므로, 작업을 시작하기 전에 아래 사항을 무조건 준수하십시오.

---

## 📌 1. 피그마 고충실도(High-Fidelity) 디자인 이식 수칙

피그마 디자인 시스템과 구현물 간의 오차를 최소화하기 위해 다음 규칙을 절대적으로 적용합니다.

### 🚫 임의의 문자열 각색 및 이모티콘 추가 절대 금지
- 피그마 텍스트 노드에 기재된 문자열을 토씨 하나 틀리지 않고 **100% 동일하게** 하드코딩하십시오.
- 친근함을 더하기 위함이라도 피그마에 없는 **이모티콘(예: `😊`, `🌿`, `🧺` 등)을 임의로 추가하는 행위를 전면 금지**합니다.
- 예: `안녕하세요! 여기선 일기를 써야 한다는 부담은 내려 놓으세요.` (피그마 원안 문자열) ➡️ 이모티콘을 덧붙이지 마십시오.

### 📏 Spacer() 남용 금지 및 밀착 레이아웃 구조화
- 반응형 화면비를 맞추기 위해 위젯 사이사이에 `Spacer()`를 무분별하게 배치하지 마십시오. 화면 해상도가 큰 기기에서 요소 사이의 간격(특히 말풍선과 캐릭터 사이)이 기형적으로 멀어지는 원인이 됩니다.
- 피그마 절대 좌표 비율을 코드에 반영할 때는 요소들 간의 간격을 **컴팩트한 고정 크기(`SizedBox(height: 24)` 등)로 촘촘히 묶어 먼저 배치**하고, 최하단의 주요 버튼 영역 등만 `Spacer()`를 활용하여 화면 맨 밑으로 밀어내도록 설계하십시오.

### 🎨 타이포그래피 및 명도(Color) 명세 엄수
- 임의로 캡션 스타일을 적용하거나 글자 크기, 색상의 명도를 낮춰서 구현하지 마십시오.
- 피그마에 명시된 폰트 크기 및 색상(예: `body2_R` ➡️ `16px`, `Color(0xFF1A1A1A)` 검은색)을 그대로 이식하십시오. 흐릿하고 어두운 회색으로 본문을 임의 해석하는 행위를 방지합니다.

## 💾 2. 에셋(Asset) 관리 및 보존 수칙

### 🛡️ 수동 다운로드 에셋의 영구 보존
- 사용자가 직접 피그마에서 추출하여 수동으로 배치한 이미지 에셋(예: `assets/images/character.png`)은 **절대로 AI 스크립트나 자동 다운로더로 덮어쓰거나 훼손하지 마십시오.**
- 동적으로 이미지를 가져오거나 에셋 파이프라인을 자동화하려다 엉뚱한 이미지가 다운로드되어 사용자의 소중한 수정 내역이 유실되는 사고를 원천 방지합니다.

### 👁️ 에셋 구조의 시각적 검증
- 로컬 피그마 companion 서버의 에셋 정보와 피그마 실시간 API 상의 컴포넌트 정보가 일치하는지 항상 더블 체크하십시오.
- 단순 프로필 영역에는 전신 마스코트 대신 기호화된 프로필 이미지(예: 테두리만 있는 원형 등)를 사용하도록 지정되어 있는지, 레이아웃의 실물 렌더링 상태를 꼼꼼히 대조하십시오.

## 📝 4. 피드백 및 트러블슈팅 히스토리 기록부

여기에 사용자의 지적 사항과 수정 이력을 간략하게 보존하여, 다음 턴의 AI가 동일한 실수를 하지 않도록 대비하십시오.

### [2026-06-01 수정 이력]
- **문제**: 1~3단계 소개 화면에 뒤로가기 버튼 노출, 말풍선 내 이모티콘 임의 추가, 말풍선과 캐릭터 사이의 비정상적으로 넓은 간격 발생, 4단계 캘린더 예시의 아바타 이미지 오류 및 설명글이 너무 작고 흐림.
- **원인**: `Spacer()`의 비효율적 배치로 인한 세로 팽창, 디자인 노드의 텍스트 임의 추가 및 스타일 해석 오류, 프로필 영역에 전신 캐릭터를 억지로 구겨 넣음.
- **해결**:
  1. `_currentPage < 3`인 경우 뒤로가기 버튼을 `null` 처리하여 숨김.
  2. 말풍선 이모티콘 완전 삭제 및 3단계 서브타이틀 멘트 원안 반영.
  3. 말풍선과 캐릭터 사이를 고정 크기로 좁게 연결하고 최하단만 `Spacer` 처리.
  4. 캘린더 예시 카드의 프로필을 피그마 명세와 일치하는 원형 프로필(`Ellipse 2`)로 바꾸고, 설명 텍스트를 `body2R` 및 `grayScale9` 검은색으로 교정하여 크기와 선명도 극대화.

### [2026-06-01 추가 피드백 및 수칙 개정]
- **문제**: 4단계 캘린더 연동 화면에서 제목 및 설명글의 가로 정렬이 정밀하게 가운데로 맞춰지지 않고 왼쪽 정렬되어 배치됨.
- **원인**: 피그마 디자인 노드의 가로 정렬(`text-center`) 명세를 누락하고 `Column`에 `crossAxisAlignment: CrossAxisAlignment.start`만 단순 기입함.
- **해결**:
  1. `_buildCalendarSyncPage`의 `Column` 정렬을 `CrossAxisAlignment.center`로 전면 교정.
  2. 타이틀 및 본문 `Text` 위젯 각각에 `textAlign: TextAlign.center`를 확실하게 부여하여 레이아웃을 완벽하게 가운데로 정비함.
  3. **사용자의 개발 제어 규칙 누적 명문화 수칙 추가**: 사용자가 AI 어시스턴트의 일하는 방식 및 룰을 교정할 때마다, 즉각 `GEMINI.md`에 내용을 요약하여 보존할 것을 규정함.
  4. **깃(Git) 조작 생략 지침 및 불필요한 안 한 것 보고(면책성 문구) 금지**: 사용자가 깃 푸시/커밋을 요청하기 전에는 관련 조작을 행하지 않되, 매 채팅 답변 시 "조작을 진행하지 않았다"는 식의 면책성 중복 보고를 금지하여 토큰 낭비를 차단함.
  5. **텍스트 행바꿈(Newline `\n`)의 시각적 크로스 체크**: 텍스트 데이터의 내용이 일치하더라도 반응형 글상자에 의해 행바꿈이 발생하는 형태가 피그마 원안과 다를 수 있으므로, 캡처된 실행 화면과 피그마 원안의 텍스트 줄바꿈 구조를 육안으로 상호 대조하여 필요시 명시적 개행 문자(`\n`)를 삽입해 100% 매칭시킬 것.
  6. **3단계 소개 말풍선 줄바꿈 오류 수정**: 3단계 말풍선 문구의 경우 피그마 원안 상 2줄 구조(`지친 하루에는 마음을 달래줄\n해결책도 알려드려요!`)였으나, 임의로 3줄 구조로 적용된 오차를 발견하여 원안에 맞추어 2줄 구조로 교정함.
  7. **마중이 캐릭터 그림자 중복 처리 제거**: 마중이 캐릭터 이미지 에셋(`character.png`) 자체에 타원형 그림자가 이미 포함되어 있으므로, 코드상으로 별도 구현되었던 `Ellipse 1` 타원형 그림자 컨테이너 위젯을 완전히 제거하여 렌더링 중복을 방지함.
  8. **이름 입력 폼의 고충실도화**: 이름 입력 필드의 배경색을 흰색(`white`)으로 설정하고 진입 시 자동 포커스(`autofocus: true`)가 동작하도록 개선함.
  9. **캐릭터 일러스트 크기 기본 비율 유지 및 마지막 단계 스케일 대응**: 온보딩 일러스트 기본 크기는 피그마 실측 규격인 160x240 (scale = 1.0)으로 원안 배치하되, 마지막 6단계인 말투 선택 페이지에서만 선택지 카드 배치 밀도를 보완하기 위해 132x198 (scale: 0.825)로 정확히 작아지도록 튜닝 적용함.
  10. **상단 타이틀 여백 일관성 강화**: 타이틀과 상단 AppBar 간의 여백을 기존 `12px`에서 `40px`로 확대 조정하여 화면 배치의 쾌적함과 디자인 시스템 상의 여백(Layout Margins)을 균형 있게 구현함.

## 🧭 5. 대규모 파일 모듈화 및 유지보수 규칙

단일 화면 내에서 다단계(Multi-step) 시퀀스를 처리하거나 파일 크기가 `500줄`을 초과하여 복잡도가 급격히 올라가는 경우, 유지보수의 영속성과 코드 품질을 위해 아래 규칙에 맞춰 모듈화를 적극 수행합니다.

### 🧱 다단계 화면의 독립적인 위젯 분리
- **원칙**: 각 단계별 온보딩 서브 뷰(예: 이름 입력, 말투 선택 등)는 `lib/screens/onboarding/widgets/` 경로 아래에 각각 개별 위젯 파일로 철저히 격리합니다.
- **네이밍 규칙**: 파일명은 `onboarding_[기능]_step.dart` 형식을 적용해 특정 화면의 종속 컴포넌트임을 한눈에 파악할 수 있도록 표준화합니다.

### 🔒 결합도 최소화 및 독립 유지보수 구조 설계
1. **의존성 주입의 콜백화(Callbacks)**:
   - 개별 스텝 위젯은 부모 `PageView`나 `PageController`에 직접적으로 의존해서는 안 됩니다.
   - 데이터 검증 완료 시 부모에게 알리기 위한 콜백(`onSubmitted`, `onChanged`) 혹은 부모의 네비게이션 트리거 함수를 파라미터로 주입받는 방식으로 설계하여 개별 컴포넌트의 단독 테스트 가능성을 유지합니다.
2. **공통 데코레이터 및 레이아웃 위젯화**:
   - 말풍선 디자인(`OnboardingBubble`), 일러스트 베이스(`OnboardingLayout`) 등 모든 단계에서 공통적으로 쓰이는 특수 시각 요소는 최상위 공통 패키지 위젯으로 별도 추출하여 중복 구현을 방지합니다.
3. **상태 관리의 캡슐화**:
   - 각 단계 내에서만 소모되는 일시적 UI 상태(예: 텍스트 필드의 포커스 여부, 애니메이션 상태 등)는 부모 스크린으로 전파하지 않고 서브 스텝 위젯 내부에서 자체적으로 완결 지어 캡슐화합니다.

### 📐 신규 로직 추가 시 모듈 분리 여부 판단 기준
새로운 기능(예: 이미지 첨부 미리보기, 파일 탐색, 바텀 시트, 다이얼로그 등)을 추가할 때, 곧바로 스크린 파일(예: `chat_screen.dart`)에 코드를 인라인으로 작성하지 않고 **모듈로 분리할지 여부**를 판단하기 위해 다음 3단계 의사결정 프로세스를 적용합니다.

#### 1단계: 모듈 분리 판단 체크리스트 (Decision Matrix)
아래 4가지 조건 중 **단 하나라도 해당**하면, 독립적인 위젯이나 헬퍼 모듈로 분리하여 구현합니다.
- [ ] **라인 수 한계선**: 구현 시 스크린 본체 파일의 전체 코드 길이가 **400~500줄**을 초과할 것으로 예상되는가?
- [ ] **자체 로컬 상태(Local State) 보유**: 해당 기능이 고유한 컨트롤러(예: `TextEditingController`, `ScrollController`)나 임시 상태(예: 선택된 이미지 경로 `_selectedImagePath`, 애니메이션 상태)를 자체적으로 가지고 있는가?
- [ ] **UI 중첩 깊이(Depth)**: 위젯 트리 내에서 해당 UI 블록의 중첩 Depth가 5단계 이상이거나 가로/세로 레이아웃이 복잡하여 전체 빌드 메서드의 가독성을 해치는가?
- [ ] **재사용성**: 해당 다이얼로그, 바텀 시트 또는 카드 뷰가 향후 다른 화면에서도 공통으로 사용될 가능성이 있는가?

#### 2단계: 모듈화 설계 시 준수 사양
모듈로 분리하기로 결정된 경우, 아래 설계 방식을 엄수합니다.
1. **의존성 차단 (Stateless & Callback)**:
   - 분리된 하위 위젯은 스크린의 특정 상태에 직접 의존하지 않고, 필요한 데이터만 매개변수로 받으며 상위 상태 변경은 콜백(`onPressed`, `onChanged`, `onCancel` 등)으로 상위 스크린에 전송합니다.
2. **폴더 구조화**:
   - 특정 화면에만 종속되는 컴포넌트는 `lib/screens/[화면명]/widgets/` 내에 생성합니다. (예: `lib/screens/chat/widgets/`)
   - 앱 전역에서 범용적으로 사용 가능한 위젯은 `lib/widgets/` 내에 생성합니다.

#### 3단계: 인라인 작성 예외 기준
상기 체크리스트에 해당하지 않고 아래 조건을 모두 충족하는 경우에만 스크린 내에 직접 인라인 코드로 작성합니다.
- 추가되는 로직이 단순한 레이아웃 구조체이며 50줄 미만인 경우.
- 화면 전체에 걸친 상태 관리 기계(State Machine)나 메인 UI 프레임워크 자체인 경우.
